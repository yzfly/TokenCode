# TokenCode 路线图

> 这张图从内核推导，不是凭空排的功能清单。当前状态见 [`STATUS.md`](STATUS.md)；每天的脚印见 [`devlog/`](devlog/)。

## 内核回顾（这张图为什么长这样）

TokenCode 要做的是「**为并行而生的运行时**」：并行是底座，agent 是底座上长出来的东西——不是给单 agent 加一个并行功能。并行有两种，两种都要：

- **A · 横向爆破（竞赛）**：同一个问题派一群 agent 各自独立解，裁判取最好的一份。冗余求质量。几乎 share-nothing。
- **B · 协作**：一群 agent 分工，拼出谁单独都拼不出的解。分工求规模。要碰共享状态。

建造顺序已定：先建你 99% 时间真正在跑的默认模式，给最难的通用 case 留缝，别预先投入。这张图就是这个顺序的展开。

## 阶段 0 · 单 agent MVP 底座 — 已完成

**目标**：用 Go 跑通最朴素的编码 agent——单 agent 的 tool-use 循环。

**做了什么**：Anthropic 协议客户端（经 DeepSeek 接入）、完整 agent loop、read/write/edit/bash 四工具、纯文本 REPL、零第三方依赖单二进制。`build`/`vet`/`test` 全绿。

**为什么先做它**：A 要同时跑成百上千个 agent，前提是先有一个可用的。这一层是**后续所有并行的底座**。

**与内核的关系**：单 agent 本身离目标还远，但承重的地基在这里。

详见 [`devlog/2026-06-10.md`](devlog/2026-06-10.md)。

## 阶段 1 · 可用、好看的交互外壳（TUI）— 已完成 / 打磨中

**目标**：把糙的纯文本 REPL 换成 Claude Code 级观感的 TUI。

**做了什么**：Bubble Tea alt-screen + viewport、glamour markdown 渲染（每条一次进缓存）、DeepSeek 蓝主题、亮/暗自适应配色、权限三模式（plan/review/yolo + Shift+Tab/slash 切换）、字符 logo + 欢迎卡、非 tty 纯文本回退。

**为什么**：这个项目要「一起来做」，得有人愿意上手——能跑、顺、好看是门槛。也借此把工具确认升级成了真正的权限模型（plan/review/yolo），为将来「一千个 agent 同时要动文件」时的授权策略练手。

**已知限制 / 打磨项**：无终端原生 scrollback（alt-screen 的取舍，滚动靠 viewport + 滚轮）；不流式；历史不持久化。详见 [`devlog/2026-06-10.md`](devlog/2026-06-10.md)。

**与内核的关系**：和并行内核**正交**——它是外壳，不是脊柱。但权限模型这一笔，是为多 agent 授权埋的伏笔。

## 阶段 2 · A · 横向爆破（竞赛）— v1 已落地（2026-06-11）· 打磨中

**目标**：在单 agent 底座上，派 N 个 agent 各自独立解同一个问题，裁判取最优，败者退钱（取消、停手、退预算）。

**为什么先于复杂的 B**：A 几乎 share-nothing，不碰共享状态，是**最快验证「大规模并行」这件事本身成立**的路径。

**v1 落地了什么**（`internal/race` + `/race` 命令）：

- `/race <N> <任务>`：N≤1000，每个 racer 一个 git worktree（独立分支、共享对象库）+ per-registry 工具根（文件工具锁死在自己 worktree、bash 在其中执行）——写空间完全隔离。
- 窗口化并发（`race.concurrency`，默认 8）+ `context` 取消顺树传导：Ctrl+C 全员停手、worktree 全部清理。
- 裁判流水线：客观粗筛（空 diff 淘汰、`race.check` 命令淘汰，零 token）→ 并行 LLM 打分 → top-4 决赛终审。
- 落地有人闸：排行榜 + 冠军 diff 摘要，`/race apply` 确认后 `git apply` 回主工作区（不自动 commit），冠军分支保留可追溯。
- 多样性扩展点（variant 生成器）已留位，v1 同 prompt 自然采样。

**开放问题的现状**：

- **裁判机制**：v1 = 客观信号 + LLM 打分 + 决赛。投票 / 多视角对抗验证待加。
- **成本与背压**：窗口压并发 + llm 层重试兜底；预算上限（钱）尚未建模。
- **上下文隔离**：已解决——racer 是全新上下文的子代理，写空间靠 worktree + 工具根硬隔离。
- **败者提前退钱**：v1 全员跑完再裁判；「第一个足够好就 cancel 全场」待做。

**与内核的关系**：从这里开始，「并行运行时」不再只是路线图上的一句话。

## 阶段 3 · B · 协作 + 工作区权威（单写者）— 未开始

**目标**：一群 agent 在同一个工作区分工干活，各自成果最后汇到一起、彼此一致（这是「甲」：逻辑上同一个代码库，不要求看见对方半截的编辑）。

**架构脊柱**：

> agent 永远不直接碰文件系统，只发消息给「工作区权威」。

- **工作区权威 = 单写者 actor**：只有它一个有权改文件。agent 把「我要打这个 patch」作为消息发给它，它排队、校验、三方合并，要么应用、要么打回（「你这 patch 基于版本 N，现在已 N+3」）。agent 只读快照。
- 因为只有一个写者，「Go 不防数据竞争」压根不发生——共享工作区不但没动摇 Go，反而坐实了 Go：整套机器（工作区 actor、租约 broker、合并编排、冲突升级）全是并发协调代码，全是 Go 的主场。

**推荐组合**（不是单选）：① 协调者先做粗粒度语义分区压低冲突率；② agent 在各自快照/worktree 上乐观并行干；③ 工作区权威串行化「提交」这一步做三方合并；④ **冲突不是失败，是信号**——真撞同几行了，交给一个「调和 agent」重解（或重跑、或升级给人）。

**开放问题**：冲突升级策略（调和 agent 的输入是三方 diff 还是双方意图？失败往哪兜底）。

**与内核的关系**：这是「Git 即合并」那个洞见的落地——一个多 agent 共享工作区的运行时，剥到底就是一个版本控制系统，只不过合并冲突不甩给人类，而是用智能去消解。
