# 2026-06-10 · 第一天：单 agent 底座跑通了

今天把 TokenCode 的地基打完了。这个项目要做的写在 README 第一行：**Token 燃烧机，为 Token 燃烧而生**——把 token 当燃料，派一群 agent 并行猛烧，用冗余换质量。但要烧得起，先得有一个能干活的单 agent，所以今天不碰并行，全部的活就是把单 agent 底座做扎实，再让它活起来。

## 先跑通最朴素的循环

agent 的核心没有魔法，就是一个循环（`internal/agent/agent.go`）：

```
append user(text)
loop:
  resp = llm.Complete(model, system, msgs, tools)
  append assistant(resp.text, resp.toolUses)
  if no toolUses: return        // 本轮结束
  for each toolUse:
    confirm? execute -> result
  append user(toolResults)      // 结果回灌
```

四个工具：`read` / `write` / `edit` / `bash`。grep、find、ls 先用 bash 顶着，够用。

为什么用 Go：agent 99% 的时间在**等**——等 LLM 回包、等命令跑完。瓶颈是 I/O 并发加消息协调，不是算力，这正好是 Go 的主场。goroutine / channel / `context` / `errgroup`，跟我后面要做的并行原语几乎一一对应。

内部协议我选了 Anthropic Messages，没选 OpenAI chat：`tool_use`/`tool_result` 的 content block 结构更干净，表达力是上界。这个选择在做多模型转换层的时候赚到了——别家协议转进来是升格，反过来才有损。

## TUI：好看，但绝对不能卡

我对 opencode 的卡顿忍了很久。这次自己写，先把根因挖出来：它卡不是因为界面形态，是因为**流式时每个 token 都把整段已累积的 markdown 重新过一遍 glamour、再整屏重绘**，消息越长越卡。

所以这版从结构上绕开：

- **每条内容只渲染一次**。新条目来了用当前宽度过一遍 glamour，结果进 `rendered` 缓存；之后每帧只是把缓存 join 起来，不重渲。只有终端宽度变化才整批重渲一次。
- **这版先不做流式**。没有「每 token 重渲染」这回事，每帧的成本就是拼字符串。

外壳形态选了 **alt-screen + viewport**：鼠标滚轮滚对话、缩放终端整屏干净重排、底部固定输入框和状态栏。代价是放弃了终端原生 scrollback 和选择复制的习惯——这是个真实取舍，将来想换 inline 模式找回原生滚动，得先解决它的 resize 残留问题。

一条铁律贯穿整个 TUI：worker goroutine 只 `program.Send` 原始数据，**所有渲染在 `Update` 单线程里做**（只有它知道当前终端宽度）。这版不卡，不是调出来的，是结构决定的。

功能都齐了：markdown 渲染、DeepSeek 蓝（`#4D6BFE`）、亮暗自适应、权限三模式（plan 只读 / review 逐次确认 / yolo 全放行，Shift+Tab 切）、字符 logo、↑↓ 历史、Ctrl-C 打断当前轮、管道模式退化纯文本。依赖上我认了 charmbracelet 这笔账——核心层还是纯标准库，外壳花钱买观感。

## 多模型：协议只有两种，模型有无数个

我不想给每个模型写一个适配器。看清楚之后其实就两层：

- **协议**：报文格式，目前两种——`anthropic` 和 `openai-chat`，每种一个 codec（`internal/llm/anthropic.go`、`openai.go`，都是纯 net/http）。
- **provider**：纯配置——base_url + key + 协议类型。Kimi、Qwen、OpenRouter、Ollama 全是 openai-chat 下面的不同 base_url。

加一个模型 = 在 `~/.config/tokencode/config.json` 加一段配置，零代码，然后 `-model 别名` 或 `-model provider/model-id` 切换。不写 config 的话行为跟默认完全一样。

转换层的坑全锁在两个纯函数和 golden 测试里，几个值得你直接抄走的细节：

- tool 结果在 Anthropic 是一条 user 消息里的多个 block，在 OpenAI 是每个结果一条独立的 `role:"tool"` 消息，必须紧跟对应的 assistant——拆错顺序两边都直接 400。
- DeepSeek/Qwen 的 `reasoning_content` 回传历史会 400。我让 `Thinking` 只存在于 `Response`、不进 `Message`，回灌历史时天然剥掉。
- tool 调用的 `arguments` 空串要补 `"{}"`；`max_tokens` 和 `max_completion_tokens` 我两个都发，新旧端点通吃。

## 心跳和做梦：让它活着，但别烧我的钱

agent 不该只在我说话的时候才醒。今天装了两件器官（`internal/pulse`）：

**心跳**（`-heartbeat 30m`，默认关——让 agent 自己花钱的开关必须人来开）。最怕的是空转烧 token，所以三级短路：先跑本地检查（工作区 mtime、todo.md），九成的拍零 token 就结束了；有动静才调一次模型，回 `HEARTBEAT_OK` 就把这一拍从历史里删掉，不污染上下文；真有事才当完整的一轮跑。

**做梦**。空闲且有新材料时触发——光空闲不行，没新料的梦是复读。后台 goroutine 调一次无工具的 LLM，把会话历史压缩重写进 `.tokencode/memory.md`，下次起来注入 system prompt。聊一段、晾十分钟、重启进程，它还记得你说过什么。做梦的本质是压缩不是堆积，所以是重写不是追加。

底下的地基是个很 Go 的设计：用户消息、心跳、梦醒，都是同一个 `Event` 的不同来源；agent 是消费 `chan Event` 的 actor，所有 turn 串行，消息历史只有一个写者。心跳和梦发起的 turn 权限从严——只读放行、写直接拒，绝不进那条等人按键的确认路径。宁可它干不了活，不能让它把自己卡死。

## 调研：Claude Code / OpenCode / Codex 各自怎么实现的

四份报告全文在 [`../docs/research/`](../docs/research/)，几条最值钱的结论：

- **三家都在向同一个原则收敛：启动只注入索引，正文按需加载。** Claude Code 做得最极致——skills 只注入 name+description（预算制），MCP 工具 schema 全部 deferred，连内置工具都 defer 了（约 15k token 降到约 1k）。这条铁律 TokenCode 第一天就得遵守。
- **Node/Bun 的启动开销是 Claude Code 官方承认的痛点**（为此推 `--bare`）；OpenCode 背着整个 Bun runtime，还有 RSS 超 2GB 自动 dump 的守护逻辑；Codex 用 Rust 拿到了毫秒启动但二进制 80MB。Go 的位置正好：毫秒启动 + 我们今天的二进制 20MB。
- **三家都没有内置 cron，OpenCode 和 Codex 都没有官方市场**——这两块加上"激进并行"就是我们的差异化阵地。
- 值得直接抄的细节我都写进 [`../docs/architecture.md`](../docs/architecture.md) 了：hooks 的退出码协议（0/2/其他）、市场=git 仓库+marketplace.json、会话 JSONL+parentUuid 链、OpenCode 用 root commit hash 当 projectID、Codex 的"同一二进制 arg0 重入做沙箱 helper"。

## 架构图：docs/architecture.md

完整设计落盘：性能预算表（冷启动 <50ms、二进制 <25MB、启动零网络 IO，CI 守护）、六条设计铁律、core+事件协议+多前端的骨架、skills→hooks→plugins→marketplace 的实现顺序、subagent 并行模型、分层 memory、内置 cron 三层、channels（飞书优先）。M1 到 M6 的里程碑排好了。

## 落地：协议层按协议命名，Google 进来了

上一篇结尾说"真要加新协议，照着 openai.go 写第三个 codec"——说到做到：

- `protocol` 现在是三个值：`anthropic`、`openai`、`google`。**模块就叫协议名**，不叫 "openai-chat" 这种含混的名字；旧值 `openai-chat` 在 Load 时静默归一化，已有 config 不破坏。

# 同日第三篇：M2 落地——流式 + 会话持久化

按架构图的里程碑顺序，把 M2 的两块都做完了。这两块都是"快"的直接体感：流式让首字延迟从"整段生成完"变成"第一个 token 到"；JSONL 持久化让 `-continue` 一秒接上昨天的活。

## 流式：一个接口，三个协议，agent 循环零改动

设计上最重要的决定：`Streamer` 是 codec 的**可选能力**，`CompleteStream` 逐段回调增量、**最终返回与 `Complete` 完全同语义的 Response**。agent 循环只多了一个三行的分支（支持流式且外壳要增量→流式，否则照旧），tool-use 循环、历史回灌、心跳剔除全都不知道流式的存在。

三个协议的事件形态完全不同，坑各自锁在各自的 codec 里：

- **anthropic**：`content_block_*` 按 index 多路复用，tool_use 的 `input_json_delta` 分片拼 JSON；`message_start` 给输入用量、`message_delta` 给输出用量和 stop_reason。
- **openai**：`delta.tool_calls` 按 index 聚合、首片带 id/name 后片只有 arguments 碎片；`[DONE]` 结尾；用量要 `stream_options.include_usage` 才给（不支持的端点静默为零）。
- **google**：每个 chunk 就是一个完整响应形态，文本逐段、functionCall 整只到；累积后直接复用非流式的组装函数，合成 ID 规则保持一致。

共用一个 30 行的 SSE 读取器（4MB 行缓冲——工具参数可能很大）。TUI 侧流式期间显示按宽度软换行的原始文本尾巴，完成后用 markdown 渲染整段替换——每个 delta 都跑一遍 glamour 会卡，这个取舍值得记住。

## 持久化：水位线，而不是边写边记

难点不在写文件，在**心跳空转拍要剔除**：如果消息一追加就落盘，哨兵拍会先写进去再想办法删——append-only 文件没有删。解法是 agent 持一条**持久化水位线**：每拍结束、剔除逻辑跑完之后，才把水位线之上的消息批量交给回调；truncate 把历史截回去时水位线跟着回落，被剔除的拍**从未**落盘。`Seed`（resume 注入）把水位线直接设到历史末尾，旧消息不会重写一遍。

文件格式照抄调研结论：`sessions/YYYY/MM/DD/<id>.jsonl`，id 内嵌日期所以由 id 可直接反推路径（不需要索引文件）；首行 meta（含 cwd），`-continue` 按 id 倒序扫、读首行比对 cwd。崩溃尾巴（写了半行）Load 时静默跳过——这是 Claude Code 真实事故（#40352）教的。

## 测过的，实话实说

- 单测：三协议流式（事件序列/分片拼接/用量）、session 往返/追加/Latest/损坏行跳过、agent 水位线（多拍不重复、空转不落盘、Seed 不重写）。`go build / vet / test -race ./...` 全绿。
- **端到端冒烟**：起假 Anthropic SSE 服务，真跑 `bin/tokencode`——流式输出逐段到、JSONL 文件内容正确、`-continue` 恢复出 "已恢复会话…（2 条消息）"。这次活体冒烟做了。
- 没测的：真端点（DeepSeek/Gemini）的活体流式；TUI 流式渲染只看了代码没上 pty。

## 想接力的话

- 流式入口在 `agent.runTurn` 的那个分支；新协议照 `google.go` 的"累积成合成响应、复用非流式组装"模式最省心。
- 持久化语义全在 `agent.flushPersist` 的注释里；动历史剔除逻辑时记得它和水位线的约定。

# 同日第四篇：TUI 命令体系——/help /model /skills /mcp + 补全菜单

用户点名要 TUI 支持 /skills /mcp /model /help。做之前先派了个调研 agent 把 Claude Code 的 help 与命令体系拆了一遍，产出 [`../docs/requirements/tui-commands.md`](../docs/requirements/tui-commands.md)（P0/P1/P2 三级需求，60+ 内置命令按四种输出形态归类）。调研出的最关键一条：**Claude Code 的命令发现主面不是 /help，是输入 / 时弹出的补全菜单**——/help 只是它的纯文本备份。所以这轮把菜单也一起做了。

## 命令注册表：三处同源

地基是 `internal/tui/commands.go` 的命令注册表：`{name, aliases, argHint, summary, source, run}`。/help 的列表、/ 补全菜单的候选、submitInput 的分发，全部从这一张表生成——新增命令只写一条记录，三处自动一致。技能（skill.Discover 的结果）作为动态命令源注入同一张表，`/技能名 [参数]` 调用时才读盘展开正文（渐进披露不破）。

交互细节照需求文档落实：未知命令**绝不发给模型**（拼错命令不烧钱），给前缀/编辑距离 ≤2 的就近建议（/exi → 是否想用 /exit？）；`//` 开头是转义，剥一个斜杠当普通消息发；补全菜单前缀优先、子串次之，↑↓ 选、Tab 补全、Enter 对无参命令直接执行、Esc 关闭（值不变不重开）。

## 这轮顺手把两块"未来件"提前落了
